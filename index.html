About prei, pret and paroa.

When you're learning Dutch as a foreigner and try to speak it with real Flemish people after a lot of sweat and tears of learning a languange with so many exceptions to the grammar rules, you soon discover that people aren't speaking Dutch at all, but Flemish, a weird unintelligble variant.

And then there's the dialects. You walk ten minutes and there's, again, an other dialect. As a native speaker I got acquainted with the phenomenon at an age of six, when my west-flemisch dialect speaking parents decided to move to the Dender region. My mom asked me to buy some leek at the grocery store. Prei (leek), pret, and paroa are the same thing, but you don't know that yet as a kid. It took a run back home and a translation from pret to prei, and later at the grocery store from prei to paroa to finally to be able to acquire with pride and determination the long sought after vegetable.

But I guess this event sharpened my intuition for the languages : French, English, German, Music, Mathematics, Basic, C, Matlab, and Java. There is more to discover about me in pictures.

On radically un-duplicating code

One of the things most scorned in the programming community, is code duplication. After some hours of creative development, you should always look critically at your code, and refactor the parts containing the same code fragments.

But when you take a very close look at your typical preferred programming language, you might discover that repetition is sometimes embedded in the core dna of your working tools. Of course, the language creators always have a very good reason for this. But sometimes you wish we didn't have to repeat ourselves. Let's have a look at some Java examples.

Any time I have to write a typical Java line as Thing thing = new Thing(); - this line almost sounds like chuch bells - my blood pressure starts rising. Although every-thing is needed here when we want to include information pertaining to polymorphism, it really hurts my eyes.

In other cases, you create a class with three fields, meant to keep related data together. According to the Java Canon, you're supposed to create getters and setters. Context decides that rudimentary getters and setters suffice for now. With complex classes, the bulk of your class might consist of a quasi endless list of three lines long get and set methods. On top of that in most cases you just need a standard constructor with the three fields, and maybe an empty constructor. The result is that you have to repeat your field name and types four times to configure your class correctly.

Some people in the Java community discovered a nice solution to some of these inherent repetitions. The Lombok project is a nice example how an "old" programming language keeps evolving, providing tools for writing code more efficiently. Do you also think this is awesome ? Let me know !
